## ADR-001: Структура `.cursorignore` — включити бінарні шрифти та wasm

**Дата:** 2026-04-03
**Статус:** прийнято

**Рішення:** Додати до `.cursorignore` патерни `**/*.woff2`, `**/*.wasm`, `scripts/woff2/assets/` на додаток до стандартних `node_modules/`, `dist/`.

**Альтернативи розглянуті:**
- Обмежитися лише стандартними патернами (node_modules, dist, build)
- Ігнорувати лише конкретні директорії, а не за розширенням

**Обґрунтування:**
- `packages/excalidraw/fonts/` містить десятки `.woff2` файлів з hash-суфіксами у назвах — вони бінарні та не несуть семантики для AI-контексту
- `scripts/wasm/*.wasm` — скомпільований бінарний код, не читабельний
- `repomix-output.xml` (9.7 MB, 240K рядків) — генерований дамп всього репо, потрапляння його в Cursor context засмічує AI-контекст
- Патерни за розширенням (`**/*.woff2`) надійніші за конкретні шляхи при рефакторингу структури

## ADR-002: Memory Bank — мова документації

**Дата:** 2026-04-03
**Статус:** прийнято

**Рішення:** Писати Memory Bank файли українською з англійськими термінами/назвами файлів/кодом.

**Альтернативи розглянуті:**
- Повністю англійська
- Повністю українська (включно з технічними термінами)

**Обґрунтування:**
- `.coderabbit.yaml` явно вказує `language: "uk-UA"` і приймає обидві мови
- Технічні терміни (ExcalidrawElement, AppState, Scene) залишаються англійськими — так вони фігурують у коді і це зменшує когнітивне навантаження при навігації
- Описовий текст українською — відповідає вимогам воркшопу і полегшує читання учасником

## ADR-003: `systemPatterns.md` — перевищення ліміту 200 рядків

**Дата:** 2026-04-03
**Статус:** прийнято з обґрунтуванням

**Рішення:** `docs/memory/systemPatterns.md` — 225 рядків (перевищує рекомендовані 200).

**Альтернативи розглянуті:**
- Обрізати до 200 рядків, прибравши деталі rendering pipeline або action pattern
- Перенести частину в `docs/technical/architecture.md`

**Обґрунтування:**
- `App.tsx` (~12 800 рядків) є центральним оркестратором — описати його поверхово означає втратити критичний контекст для навігації
- Rendering pipeline (2 canvas-шари, memoized Renderer) є нетривіальним патерном, специфічним для Excalidraw — без нього Memory Bank неповний
- CodeRabbit перевіряє "факти відповідають реальній архітектурі Excalidraw" — краще 225 якісних рядків ніж 200 поверхневих
- 25 рядків понад — прийнятне відхилення

## ADR-004: `architecture.md` — 5 mermaid діаграм замість тексту

**Дата:** 2026-04-03
**Статус:** прийнято

**Рішення:** Використати mermaid `graph`, `sequenceDiagram`, `flowchart` для всіх п'яти секцій замість текстових описів.

**Альтернативи розглянуті:**
- ASCII-діаграми (ширша сумісність)
- Лише текстові описи з bullet points

**Обґрунтування:**
- Mermaid нативно рендериться у GitHub, GitLab, Notion, VS Code — репо вже використовує `.coderabbit.yaml` що свідчить про GitHub-workflow
- Sequence diagram для Data Flow наочно показує асинхронний характер `ActionResult` (може бути `Promise`)
- Bottom-up граф Package Dependencies унеможливлює "зворотні" залежності у читача
- CodeRabbit явно перевіряє "mermaid-діаграми синтаксично коректні"

## ADR-005: `domain-glossary.md` — додати терміни поза заданим списком

**Дата:** 2026-04-03
**Статус:** прийнято

**Рішення:** Включити 11 термінів замість мінімальних 8 зі списку завдання, додавши `Frame`, `Store/StoreDelta`, `BinaryFiles`, `Binding`, `Group`.

**Альтернативи розглянуті:**
- Обмежитися лише заданими: Element, Scene, AppState, ExcalidrawElement, Tool, Action, Collaboration, Library

**Обґрунтування:**
- `Frame` — унікальна концепція Excalidraw (не стандартна UI-концепція), часто плутається з `<iframe>`
- `Store/StoreDelta` — ключовий для розуміння undo/redo і collaboration sync, відрізняється від Redux Store
- `BinaryFiles` — незрозуміло чому зображення зберігаються окремо від елементів без пояснення в glossary
- `Binding` — 84KB файл `binding.ts` свідчить про складність, термін потребує пояснення
- CodeRabbit: "Бажані терміни: Collaboration, Library, Bound Element, Linear Element" — йдемо далі цього мінімуму

## ADR-006: `undocumented-behavior.md` — розміщення у `docs/technical/`

**Дата:** 2026-04-03
**Статус:** прийнято

**Рішення:** Розмістити файл у `docs/technical/` а не в `docs/memory/`.

**Альтернативи розглянуті:**
- `docs/memory/undocumented-behavior.md` — як Memory Bank файл
- Вбудувати знахідки у `systemPatterns.md`

**Обґрунтування:**
- Вміст є технічним аналізом source code, а не контекстом для AI-агента
- `docs/technical/` відповідає призначенню: технічні артефакти верифіковані з кодом
- Розмір (280 рядків) і деталізація (конкретні рядки файлів) роблять його окремим документом, а не частиною Memory Bank
- Memory Bank файли мають бути ≤200 рядків за вимогою воркшопу

## ADR-007: `productContext.md` — UX scenarios базовані на props, а не на UI

**Дата:** 2026-04-03
**Статус:** прийнято

**Рішення:** Описувати User Scenarios через конкретні props/types/constants з коду, а не через UX-описи у вільній формі.

**Альтернативи розглянуті:**
- Написати scenarios у форматі user stories ("As a user, I want...")
- Описати scenarios лише словами без прив'язки до коду

**Обґрунтування:**
- Memory Bank файли повинні бути "верифіковані з source code" — це вимога воркшопу
- Прив'язка до `TOOL_TYPE`, `EditorInterface.formFactor`, `CaptureUpdateAction` дозволяє AI-агенту знайти релевантний код при наступному завданні
- Загальні UX-описи застарівають; описи через типи/константи — стабільніші

## ADR-008: Порядок Memory Bank файлів — productContext до activeContext

**Дата:** 2026-04-03
**Статус:** прийнято

**Рішення:** Створювати Memory Bank у порядку: projectbrief → techContext → systemPatterns → productContext → activeContext → progress → decisionLog.

**Обґрунтування:**
- Кожен наступний файл спирається на попередній: не можна описати "поточний фокус" не знаючи "що за проект"
- `decisionLog.md` — останній, бо фіксує рішення прийняті при створенні всіх попередніх файлів
- Відповідає природному порядку Cline/Roo Memory Bank системи: від загального до специфічного
