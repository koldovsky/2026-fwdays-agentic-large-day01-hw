> Reverse-engineered Product Requirements Document на основі аналізу source code,

---

## 1. Проблема та мета

### Проблема

Існуючі інструменти для діаграм (Visio, Lucidchart, draw.io) вирішують задачу
"зробити красиво", але створюють психологічний бар'єр: схема виглядає як фінальний
документ, тому люди бояться її змінювати або показувати незавершеною.

Легкі альтернативи (папір, whiteboard) не зберігаються, не шеряться і не підтримують
колаборацію.

### Мета продукту

Дати найшвидший шлях від думки до спільної схеми — без інсталяції, без акаунту,
без страху зробити "не так". Hand-drawn стиль сигналізує: це чернетка, її можна міняти.

## 2. Цільова аудиторія

### Кінцеві користувачі (excalidraw.com)

**Розробники та дизайнери** — швидко замалювати архітектуру системи, flow, wireframe.
Основний сценарій: "треба пояснити колезі ідею за 5 хвилин".

**Команди** — спільна робота на одній дошці в реальному часі під час зідзвону або
асинхронно.

**Студенти та викладачі** — пояснення алгоритмів, структур даних, математичних концепцій.

### Розробники (npm-пакет `@excalidraw/excalidraw`)

Продукти, що хочуть вбудувати canvas-редактор: Notion-подібні застосунки, проєктні
інструменти, навчальні платформи. Отримують готовий React-компонент без потреби
реалізовувати canvas-рендеринг самостійно.

## 3. Продуктові вимоги

### 3.1 Canvas і елементи

**P0 — критично**

| Вимога | Деталі |
|---|---|
| Infinite canvas | Без меж, без сторінок, необмежений простір |
| Основні фігури | Rectangle, ellipse, diamond, arrow, line, freedraw, text, image |
| Hand-drawn стиль | Всі фігури рендеряться через RoughJS — виглядають намальованими від руки |
| Виділення та переміщення | Single select, multi-select (drag), переміщення, resize, rotate |
| Undo / Redo | Повна підтримка, delta-based (не snapshot) |
| Zoom та scroll | Колесо миші, pinch-to-zoom на touch, keyboard shortcuts |
| Групування | Групи елементів, вкладені групи |

**P1 — важливо**

| Вимога | Деталі |
|---|---|
| Frames | Named containers з clipping при рендерингу та експорті |
| Стрілки з прив'язкою | Arrow binding до елементів, автоматичне перебудування при переміщенні |
| Elbow arrows | Маршрутизовані стрілки з прямими кутами |
| Текст у контейнері | Текст всередині фігури, auto-resize контейнера |
| Freedraw | Згладжені лінії від руки через perfect-freehand |
| Snap to grid | Прив'язка до сітки, snap до інших елементів |

**P2 — бажано**

| Вимога | Деталі |
|---|---|
| Lasso selection | Виділення довільної форми |
| Embeddable/Iframe | Вбудовування зовнішніх ресурсів у canvas |
| Laser pointer | Анімований вказівник для презентацій |
| Magic frames | AI-генерація вмісту всередині фрейму |

## 4. Нефункціональні вимоги

### Продуктивність

- Canvas перемальовується тільки при зміні елементів (static) або pointer events (interactive)
- `throttleRAF` — не більше одного рендеру на animation frame
- `ShapeCache` — кешування RoughJS-фігур, уникає повторних обчислень
- Infinite canvas без деградації при 1000+ елементів

### Надійність

- Автозбереження в `localStorage` — втрата даних неможлива при закритті вкладки
- E2E шифрування — сервер не зберігає розшифровані дані
- PWA Service Worker — офлайн-доступ до останнього стану

### Сумісність

- Підтримка сучасних браузерів (Chrome, Firefox, Safari, Edge)
- Touch-підтримка (мобільні, планшети) — окрема обробка pointer events
- React 17 / 18 / 19 для npm-пакету

### Розширюваність (для розробників)

- `LibraryPersistenceAdapter` — кастомне сховище для бібліотеки
- `renderTopRightUI`, `renderSidebar` — слоти для кастомного UI
- `generateIdForType` — кастомна генерація ID елементів

## 5. Відомі обмеження та свідомі trade-offs

| Обмеження | Причина |
|---|---|
| Колаборація тільки на excalidraw.com | npm-пакет не нав'язує конкретний backend |
| Hand-drawn стиль незмінний | Це ключова диференціація продукту |
| Один активний інструмент за раз | Спрощує mental model для користувача |
| Немає версіонування / history між сесіями | Фокус на простоті, не на enterprise-функціях |
| PNG/SVG експорт без шарів | Excalidraw — не замінник Figma |

## 6. Метрики успіху

> Виведені з архітектури та функціоналу, не з офіційних даних команди.

| Метрика | Чому важлива |
|---|---|
| Time-to-first-draw | Скільки секунд від відкриття до першого елементу на canvas |
| Collab session length | Чи використовують колаборацію активно |
| Export rate | Чи доходять до результату (PNG/SVG/share) |
| Library usage | Чи повертаються для повторного використання елементів |
| npm downloads | Здоров'я екосистеми для розробників |
