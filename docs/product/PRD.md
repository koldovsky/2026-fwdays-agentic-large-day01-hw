# Product Requirements Document (PRD)

**Продукт:** Excalidraw (монорепозиторій `excalidraw-monorepo`)  
**Версія документа:** 1.0  
**Джерела:** `docs/memory/projectbrief.md`, `docs/technical/architecture.md`, `docs/product/domain-glossary.md`

---

## Product Purpose

Excalidraw — це веб-додаток для малювання діаграм і скетчів у стилі «від руки» та екосистема **npm-пакетів**, що надають логіку редактора й React-компонент `@excalidraw/excalidraw` для вбудовування в інші застосунки.

**Основні цілі продукту (з коду та архітектури):**

1. **Запуск повноцінного веб-застосунку** у каталозі `excalidraw-app` (Vite + React) як готового хоста редактора з усіма можливостями UI та інтеграцій (зокрема колаборації та AI-сценаріїв у застосунку).
2. **Модульна поставка домену** через пакети з чіткими межами відповідальності: `@excalidraw/common`, `@excalidraw/math`, `@excalidraw/element`, `@excalidraw/utils`, `@excalidraw/excalidraw` — щоб інтегратори могли збирати й оновлювати частини незалежно.
3. **Передбачуваний потік даних** від вводу (події, гарячі клавіші, імперативний API) до зміни агрегату сцени (`Scene`), знімків `Store`, історії `History` та оновлення полотен — без суперечностей між «канонічним» станом елементів і UI-станом (`AppState`).

Продукт орієнтований на **швидке прототипування візуальних ідей**, **діаграми та схеми** з природною рукописною естетикою та на **вбудовування редактора** в сторонні продукти через публічний `ExcalidrawImperativeAPI` і колбеки зміни сцени.

## Target Audience

| Сегмент | Потреби | Як продукт відповідає |
|--------|---------|------------------------|
| **Розробники та продуктові команди** | Вбудувати whiteboard/diagram editor у свій застосунок без повторної реалізації рендерингу, історії та домену елементів | Пакет `@excalidraw/excalidraw`, приклади в `examples/*`, імперативний API (`updateScene`, `applyDeltas`, `resetScene` тощо) |
| **Дизайнери, аналітики, технічні письменники** | Швидко накреслити схеми, фрейми, зв’язки між об’єктами; експорт і спільна робота | Інструменти малювання, `ExcalidrawFrameElement`, біндинги стрілок (`FixedPointBinding`, `BindMode`), колаборація через `Collab` / `Portal` у застосунку |
| **Команди, що працюють з AI-діаграмами** | Генерація діаграм з тексту, сесії Text-to-Diagram | `TTDDialog`, `TTDPersistenceAdapter`, стрімінг `TTDStreamFetch` у зв’язці з `excalidraw-app` (наприклад `AI.tsx`) |
| **Інтегратори бібліотек шаблонів** | Повторно використовувати набори фігур | Доменний сервіс `Library` та сутності `LibraryItem` |

Аудиторія **не** обмежується одним сценарієм: і хост-застосунок, і бібліотека адресують різні ролі — від кінцевого користувача полотна до автора вбудованого рішення.

## Key Features

Нижче — ключові можливості, узгоджені з глосарієм і архітектурою (мінімум п’ять).

### 1. Редактор на полотні з шаровим рендерингом

Три рівні canvas: `StaticCanvas` (постійна сцена, сітка, фон), `NewElementCanvas` (попередній перегляд нового елемента), `InteractiveCanvas` (виділення, ручки, віддалені курсори, оверлеї). `Renderer.getRenderableElements` співпрацює з `Scene` і видимою областю перегляду (`viewport`), щоб зменшити зайве перемальовування.

### 2. Домен елементів і агрегат `Scene`

Канонічний стан малюваних об’єктів тримається в `Scene` (включно з картами елементів, фреймами та оновленнями `sceneNonce`). Елементи моделюються як поліморфні `ExcalidrawElement`; фрейми — `ExcalidrawFrameElement`; для AI-сценаріїв існує окремий тип `ExcalidrawMagicFrameElement`.

### 3. Команди редактора (`Action`) та `ActionManager`

Поведінка інкапсульована в `Action` з результатом `ActionResult`. `ActionManager` реєструє дії, обробляє гарячі клавіші (`keyTest`, `keyPriority`) і виконує команди через узгоджений шлях оновлення в `App` (`syncActionResult`).

### 4. `Store`, знімки та історія Undo/Redo

`Store` формує durable та ephemeral інкременти; `CaptureUpdateAction` визначає політику потрапляння змін до історії. `History` керує стеками `HistoryDelta` для undo/redo з узгодженням зі знімками сцени та спостережуваним підмножиною `AppState` (`ObservedAppState`).

### 5. Імперативний API для хоста (`ExcalidrawImperativeAPI`)

Інтегратори можуть програмно оновлювати сцену (`updateScene`, `mutateElement`, `addFiles`, `resetScene`, `applyDeltas` тощо), не обмежуючись лише React-props. Це окремий публічний контракт від внутрішніх методів класу `App`.

### 6. Стан редактора та політики збереження (`AppState`)

`AppState` охоплює інструменти, виділення, viewport, тему, опції рендерингу та прапорці колаборації. `APP_STATE_STORAGE_CONF` задає, які поля допустимі для браузера, експорту чи сервера — щоб уникнути витоку зайвих даних у збереження.

### 7. Колаборація та присутність (у застосунку)

Модель `Collaborator` описує віддаленого учасника; `Collab` оркеструє життєвий цикл кімнати та синхронізацію; `Portal` відповідає за транспорт (broadcast, presence, черги файлів). Інтерактивний шар рендерить віддалені курсори та виділення.

### 8. Локальні дані та бібліотека

`LocalData` розподіляє збереження між localStorage та IndexedDB для даних застосунку. Окремо доменний `Library` керує бібліотекою повторно використовуваних наборів фігур (`LibraryItem`).

### 9. Text-to-Diagram (TTD) та стійке збереження сесій

`TTDDialog` надає робочий процес text-to-diagram; `TTDPersistenceAdapter` абстрагує збереження сесій, з конкретною реалізацією `TTDIndexedDBAdapter` у застосунку. Стрімінг відповідей — через `TTDStreamFetch` (SSE, облік rate-limit сигналів).

### 10. Геометрія та прив’язки

Пакет `@excalidraw/math` підтримує примітиви для інтерактивного рендерингу; стрілки та конектори використовують `FixedPointBinding` та `BindMode` для передбачуваної поведінки прив’язки до елементів.

## Non-goals / Constraints

**Що цей PRD / продукт явно не вважає обов’язковою метою в рамках описаних артефактів:**

- **Навчальне ДЗ або зовнішній README курсу** не є джерелом істини для вимог: у репозиторії немає кореневого `README.md` з описом завдання курсу; назва локальної папки на диску не використовується як специфікація (див. `projectbrief.md`).
- **Повна специфікація всіх сторонніх сервісів** (наприклад, детальний контракт хмарного бекенду для AI) — поза межами цього документа; наведені лише доменні та інтеграційні вузли (`TTDStreamFetch`, адаптери збереження).
- **Заміна загального графічного рушія або game engine** — `Scene` у глосарії — це доменний агрегат Excalidraw, а не абстракція довільного 3D/ігрового «scene graph».

**Технічні та архітектурні обмеження (з коду):**

- Монорепозиторій на **Yarn v1** (`packageManager: yarn@1.22.22`); workspace охоплює `excalidraw-app`, `packages/*`, `examples/*`.
- Порядок збірки пакетів зафіксований скриптом `build:packages`: **common → math → element → excalidraw**; залежності між пакетами мають відповідати цьому напрямку.
- `Store` — це **не** шар персистентності в сенсі БД: для локального збереження відповідають `LocalData`, для TTD-сесій — окремі адаптери; не плутати з npm-пакетом «library» — доменна `Library` стосується бібліотеки фігур у редакторі.
- Розділення **UI-стану** (`AppState`) та **серіалізованих підмножин** — обов’язкове для коректного експорту та збереження; повний `AppState` не ототожнюється з `ObservedAppState`.
- **Portal** у глосарії — транспорт колаборації, а не React Portal для DOM — щоб уникнути плутанини при проєктуванні інтеграцій.

## Примітка щодо узгодженості

Цей PRD узгоджує продуктову мову з **DDD-орієнтованим** словником: агрегат `Scene`, сервіси `ActionManager`, `Store`, `History`, value objects на кшталт `Collaborator` та політики `APP_STATE_STORAGE_CONF`. Детальні послідовності викликів і діаграми — у `docs/technical/architecture.md`.
