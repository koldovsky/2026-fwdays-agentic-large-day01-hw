# Technical decision log — undocumented behaviors

This log records **observed code behaviors** that are **missing, vague, or inconsistent** relative to `docs/product/PRD.md`, package `README.md` files, `docs/technical/architecture.md`, and other repo documentation. It is intended for PM and engineering alignment on specs and runbooks.

| Behavior | Observed in Code | Documented Behavior | Discrepancy / Recommendation |
|----------|------------------|---------------------|------------------------------|
| Same-origin self-embedding hard block | `excalidraw-app/App.tsx`: `isSelfEmbedding` (lines 189–200) compares `document.referrer` and `window.location` origins; when true, render returns a full-screen message **"I'm not a pretzel!"** (lines 846–862) instead of the app. | PRD §4 notes collaboration is disabled in an iframe via `isRunningInIframe()`; it does **not** describe this separate same-origin self-embed guard or user-facing copy. `packages/excalidraw/README.md` does not mention embedding the hosted app in an iframe on the same origin. | **Gap:** Product/UX and support docs should state when the shell intentionally refuses to load (vs. collab-only disable). Consider linking to security/abuse rationale if this stays. |
| Global render-throttling switch | `excalidraw-app/App.tsx` sets `window.EXCALIDRAW_THROTTLE_RENDER = true` (line 155). `packages/excalidraw/reactUtils.ts` `isRenderThrottlingEnabled` (lines 34–62) returns true only when that flag is `true` **and** React ≥ 18; otherwise logs a one-time `console.warn` and disables throttling. | PRD §6–7 cite `throttleRAF` on static canvas rendering and general debouncing; `architecture.md` does not document this **window-level** integration flag or the React version gate. Library README does not mention the global. | **Gap:** Document for host app and library maintainers: purpose of `EXCALIDRAW_THROTTLE_RENDER`, when to set/unset, and React 18 requirement. |
| Share backend env vars unset | `excalidraw-app/data/index.ts`: `BACKEND_V2_GET` / `BACKEND_V2_POST` from `import.meta.env` (lines 65–66). `importFromBackend` calls `fetch(\`${BACKEND_V2_GET}${id}\`)` (line 207). `exportToBackend` calls `fetch(BACKEND_V2_POST, …)` (line 276). No guard when values are missing. | PRD §6 states behavior when backend URLs are unset is **not fully documented**; `dev-setup.md` lists env keys but not runtime failure modes. | **Gap:** Specify expected behavior (disable UI? show error? no-op?) and document that missing values yield invalid `fetch` URLs / failed exports and imports. |
| Collaboration link room-key length check | `excalidraw-app/data/index.ts` `getCollaborationLinkData` (lines 138–146): if regex matches but `match[2].length !== 22`, shows `window.alert` via `t("alerts.invalidEncryptionKey")` and returns `null`. | PRD §5 describes `#room=roomId,roomKey` and points to `getCollaborationLinkData`; it does **not** state the **22-character** key constraint or the alert behavior. | **Gap:** Add to PRD or user-facing help: valid key shape/length and what users see when a link is malformed. |
| `aiEnabled` default is on | `packages/excalidraw/index.tsx` passes `aiEnabled={aiEnabled !== false}` into `App` (line 208), so omitting the prop enables AI/TTD UI paths; `undefined` is treated like `true`. | `packages/excalidraw/README.md` does not document `aiEnabled` or its default. PRD §8 notes TTD/AI exist but not the **default-on** contract for embedders. | **Gap:** README / API docs should state: pass `aiEnabled={false}` to disable; omitting the prop leaves features visible (subject to other UI flags). |
| `#url=` hash opens remote scenes | `excalidraw-app/App.tsx` `initializeScene`: `externalUrlMatch` on `#url=(.*)` (lines 230, 303–325); `fetch(decodeURIComponent(url))`, `loadFromBlob`, optional overwrite confirm; errors set `appState.errorMessage` with `t("alerts.invalidSceneUrl")`. | PRD §5 details local load, `#json=…` share import, and `#room=…` collab; it does **not** describe this **`#url=`** deep-link flow or error surface. | **Gap:** Document supported URL/hash schemes for the hosted app (security note: arbitrary URL fetch). |
| Hash-change scene restore differs from initial load | `excalidraw-app/App.tsx`: initial `initializeScene` uses `restoreElements(..., { repairBindings: true, deleteInvisibleElements: true })` (lines 242–244). `onHashChange` path uses `restoreElements(..., { repairBindings: true })` only (lines 548–551)—**no** `deleteInvisibleElements`. | PRD §5 describes hash-driven re-init at a high level; neither PRD nor `architecture.md` calls out this **option mismatch** between first load and subsequent hash changes. | **Gap:** Either align options (if unintentional) or document why hash navigation preserves invisible elements that initial load would delete. |
